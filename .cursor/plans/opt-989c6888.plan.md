# Fix "Fake Success" - JSON Truncation Repair

## Problem

The AI Assistant fails to generate proposals because the JSON response is being truncated by the provider's token limit (likely 4096 tokens). This causes `JSON.parse` to fail, and the fallback mechanism drops the incomplete `proposal` object, resulting in "Fake Success" (reply text but no action).

## Root Cause

1. **Truncation**: The response stops mid-stream (e.g., inside `mechanics` or `lore`), leaving invalid JSON.
2. **Strict Parsing**: The current `extractJSON` utility requires complete, balanced JSON and fails on truncated strings.
3. **Schema Order**: The current schema puts `reply` first, pushing the critical `proposal` data to the end where it's most likely to be cut off.

## Solution Plan

### 1. Enhance JSON Validation (`json-validation.ts`)

- Implement `repairTruncatedJSON(json)`: A robust heuristic parser that:
- Closes unclosed strings (`"`)
- Removes trailing commas
- Closes unclosed arrays (`]`) and objects (`}`) based on a stack
- Update `extractJSON` to use this repair logic if standard parsing fails.
- Add `extractPotentialJSON`: A lenient extractor that captures JSON blocks even if they are missing their closing braces.

### 2. Optimize Prompt Structure (`assistant-service.ts`)

- **Reorder Schema**: Change the required JSON format to put `"proposal"` **FIRST**, followed by `"reply"`.
- *Old*: `{ "reply": "...", "proposal": {...} }`
- *New*: `{ "proposal": {...}, "reply": "..." }`
- *Benefit*: Truncation will cut off the `reply` (which is dispensable) instead of the `proposal`.
- **Limit Reply Length**: Explicitly instruct the AI to keep the `reply` field under 2 sentences to save tokens.

### 3. Verification

- Confirm that truncated responses are successfully repaired and parsed as valid proposals.
- Verify that the "Fake Success" logs (Has Proposal: false) are replaced with successful proposal creation.
